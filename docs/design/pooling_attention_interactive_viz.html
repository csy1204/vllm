vLLM Pooling / Reranker / Embedding Attention 시각화

단계별로 보면서, 왜 chat/decode 중심 파이프라인보다 pooling/reranker 경로가 더 빠르고 효율적일 수 있는지 비교합니다.

Step Navigator

핵심 메시지
  • Decode는 토큰 생성 루프 + KV 캐시 유지 비용이 큼
  • Pooling은 encoder-only + 집계 중심으로 경량화 가능
  • Late-interaction은 query 재활용으로 rerank 처리량 상승

1) End-to-End: Scoring/Reranking 요청 흐름

flowchart LR A[Client Request
score / rerank / embed] --> B[ServingScores] B --> C{flash late-interaction?} C -- no --> D[Standard Pooling Path] C -- yes --> E[Stage1: Encode Query
Cache token embeddings] E --> F[Stage2: Encode Docs
Return scalar score] D --> G[SequencePooler] F --> G G --> H[Response: score/relevance]

2) Encoder-only Attention vs Decode Attention

Pooling/Reranker (Encoder-only)

  • causal=False 양방향 attention
  • KV cache spec 불필요
  • 출력 후 pool/head로 바로 집계

Chat/Decode

  • Auto-regressive 토큰 루프
  • KV cache 관리/이동/압축 비용
  • step-by-step latency 누적
flowchart TD X[Hidden States] --> Y[Encoder-only Attention] Y --> Z[SequencePooler Method] Z --> W[Embedding/Classification Head] W --> O[Vector / Score]

3) Late Interaction(MaxSim) 계산 단계

flowchart LR Q[Query token embeddings] --> M[Q x D^T] D[Doc token embeddings] --> M M --> K[Mask invalid doc tokens] K --> T[Max over doc tokens
per query token] T --> S[Sum over query tokens] S --> R[Final relevance score]

핵심은 query embedding을 cache해서 여러 doc에 재사용하는 점입니다.

4) CUDA/Triton 최적화 포인트

5) 왜 더 좋고 빠른가 — 설득형 비교

기존 decode 중심 접근

Higher

  • 토큰 생성 루프 latency
  • KV cache lifespan 길어짐
  • Long context에서 memory pressure 증가

pooling/reranker 최적 경로

Lower

  • 비생성(task) 특화 경로로 불필요 decode 제거
  • encoder-only attention + pool 집계
  • late-interaction query 재사용으로 docs 처리량 증가

강조 포인트 “동일 GPU 시간에서 더 많은 문서 비교를 처리”하는 것이 reranking/embedding 서비스의 경쟁력입니다.
